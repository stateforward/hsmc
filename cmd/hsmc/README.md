# hsmc

`hsmc` is the Go-based HSM compiler for translating StateForward HSM DSL models between supported implementation languages.

The compiler owns model structure. It parses source code with tree-sitter, lowers the HSM DSL into a canonical IR, validates that IR, attributes non-HSM imports to mutable code regions as adapter context, optionally calls an adapter to translate code bodies, and emits the target language.

## Parser Runtime

The compiler frontends are intentionally tree-sitter based. The current Go implementation uses native Go tree-sitter bindings for C#, C++, Dart, Go, Java, JavaScript, TypeScript, Python, Rust, and Zig grammars.

For release distribution, the parser layer should remain an implementation detail behind each frontend. Native grammars are useful while developing, but WASM tree-sitter grammars are the preferred direction for a no-cgo cross-platform CLI because the rest of the pipeline only depends on lowered HSM IR:

```text
tree-sitter parser runtime -> language frontend -> canonical HSM IR -> validator -> adapter -> backend
```

Adapters and backends must not depend on the parser runtime. A frontend may switch from native tree-sitter grammars to WASM grammars without changing the adapter protocol, IR validation, import attribution, or target emitters.

The other names are CLI aliases. Every supported source can compile to every supported target. Behavior bodies that have not been translated to an implementation target are preserved as target-language comments inside the generated target behavior body, followed by the minimal target-language no-op or default return needed to keep the output editable. Globals that have not been translated to an implementation target are preserved as target-language comments near the top of the generated output. Diagram targets preserve captured behavior/global source as Mermaid or PlantUML notes attached to the relevant states or model.

JSON IR is a transport format. A JSON IR input must include `source_language` set to the original implementation language, not `json-ir`, so the compiler can preserve source import context and tell adapters what language behavior/global code came from.

Adapters translate mutable behavior/global code for implementation-language targets only. Compiling to `json-ir`, `mermaid`, or `plantuml` is adapter-free because those outputs are compiler-owned transport or documentation artifacts, not executable behavior languages.

## CLI

```sh
hsmc -from go -to typescript -in door.go -out door.ts
hsmc -in door.go -out door.ts
hsmc -from py -to go -in door.py
hsmc -from js -to json-ir -in door.js
hsmc -in ./machines -out ./generated/python -to python
hsmc -from go -to mermaid -in ./machines -out ./generated/diagrams
hsmc -list-languages
hsmc -list-source-languages
hsmc -list-target-languages
hsmc -list-adapters
hsmc -version
hsmc -from go -to python -in door.go -print-adapter-protocol
cat door.go | hsmc -from go -to typescript -in -
```

When `-from` is omitted, `hsmc` infers the source language from `-in` when the input path has a known extension. When `-to` is omitted, `hsmc` infers the target language from `-out` when an output path is present. If inference is not possible, the compiler falls back to Go. Use explicit `-from` and `-to` for stdin, extensionless paths, or whenever the file extension is ambiguous.

Known extensions include `.cs`, `.csx`, `.c++`, `.cc`, `.cpp`, `.cppm`, `.cxx`, `.h`, `.h++`, `.hh`, `.hpp`, `.hxx`, `.inl`, `.ipp`, `.ixx`, `.mpp`, `.tpp`, `.dart`, `.go`, `.java`, `.js`, `.jsx`, `.mjs`, `.cjs`, `.hsm.json`, `.json`, `.mmd`, `.mermaid`, `.puml`, `.plantuml`, `.py`, `.rs`, `.ts`, `.tsx`, `.mts`, `.cts`, and `.zig`.

When `-in` is a directory, `hsmc` recursively compiles every supported source file below that directory and writes outputs below the required `-out` directory while preserving relative paths. The target extension is selected from `-to`, such as `.py`, `.ts`, `.xstate.ts`, `.hsm.json`, `.mmd`, or `.puml`. If `-from` is omitted, each file's source language is inferred independently; if `-from` is present, only matching source files are compiled. Unsupported files are skipped. Common generated or dependency directories such as `.git`, `node_modules`, `vendor`, `dist`, and `build` are skipped, and an output directory nested below the input directory is not walked as input. Directory input does not support `-print-adapter-protocol`.

Use `-in -` to read source from stdin. `-out` is optional; when omitted, `hsmc` writes to stdout.

Use the Codex adapter by pointing `hsmc` at the Codex CLI. The compiler passes the adapter protocol JSON on stdin and reads the final Codex message as a `BehaviorPatch` JSON object:

```sh
hsmc \
  -from go \
  -to python \
  -adapter codex \
  -adapter-command codex \
  -in door.go \
  -out door.py
```

If `-adapter-command` is omitted, the adapter runs `codex` from `PATH`. The built-in Codex adapter invokes Codex non-interactively as `codex exec [adapter args] --output-last-message <temp-file> -`, passing the adapter protocol JSON on stdin and parsing the temp-file response before falling back to stdout. Additional `codex exec` arguments can be passed with `-adapter-args`, using a comma-separated list.

For another adapter binary that speaks the same JSON protocol, use the generic command adapter:

```sh
hsmc \
  -from go \
  -to typescript \
  -adapter command \
  -adapter-command ./my-hsm-adapter \
  -in door.go
```

Use `-print-adapter-protocol` to emit the exact JSON envelope that would be sent to an adapter without invoking one. This is useful when developing or debugging Codex and command-compatible adapters. When printing the protocol, `-out` names the protocol JSON artifact; if `-to` is omitted and `-out` has a JSON/JSON-IR extension, the target defaults to Go instead of treating the protocol artifact as a JSON IR target.

## Compiler Boundary

`hsmc` is a structural HSM compiler, not a semantic compiler for arbitrary host-language behavior bodies. The compiler owns model extraction, validation, signatures, runtime wiring, and target emission. Adapters own only semantic-code translation for behavior/global bodies and imports needed by that translated code.

Compiler-owned IR:

- models
- states and pseudostates
- initial transitions
- transitions
- triggers
- event declarations and typed event references
- guards/effects references
- state behavior references
- attributes and operations
- string operation references in `Entry`, `Exit`, `Activity`, `Effect`, and `Guard`
- deferrals
- source and target paths

Runtime configuration DSL declarations such as `Config`, `Queue`, and `Clock` are compiler-recognized runtime facilities, not compiler-owned model IR. Frontends preserve them as `Program.Globals` so adapters can translate them when requested. With no adapter, foreign-language runtime configuration is emitted as target-language comments near the top of the generated file for manual porting.

Group operation declarations such as `MakeGroup` reference machine instances but do not mutate the model IR. Frontends preserve top-level group declarations as `Program.Globals` so adapters can translate them when requested. With no adapter, foreign-language group declarations are emitted as target-language comments for manual porting.

Type utility declarations such as `MakeKind` and `IsKind` are compiler-recognized utility code, not model IR. Frontends preserve top-level type utility declarations as `Program.Globals` so adapters can translate them when requested. With no adapter, foreign-language type utility declarations are emitted as target-language comments for manual porting.

Runtime helper declarations such as snapshot helpers using `TakeSnapshot` are runtime/tooling code, not model IR. Frontends preserve top-level helper declarations as `Program.Globals` so adapters can translate them when requested. With no adapter, foreign-language helper declarations are emitted as target-language comments for manual porting.

Adapter-owned mutable regions:

- target-language `Program.Imports` returned in a patch for translated code
- `Program.Globals` entries by existing `id`; the `id` is only a selector and cannot be renamed
- target-language helper globals whose IDs start with `adapter_global_` and then use only letters, digits, and underscores
- `Program.Behaviors` entries by existing `id`; the `id` is only a selector and cannot be renamed

Adapters must not create, remove, rename, reorder, or modify model nodes. The compiler enforces behavior identity and call contract fields after patching, including `OwnerID`, `Kind`, `TriggerKind`, `Inline`, `SourceLanguage`, `Signature`, and `TargetABI`.

Top-level imports in the request are source context, not editable source structure. Patched imports must either omit `language` or use the requested target language; the compiler canonicalizes supported aliases such as `ts` or `py`, then rejects patches that claim a different import language. Behavior `target_language` and global `language` are compiler-owned and must not be returned by adapters. A behavior or global patch claims target-language ownership when it includes target body/code, unless the region was already in the target language. Import-only global/behavior patches are accepted only for code that is already in the target language; adapters should omit foreign globals or behaviors they cannot translate.

Patched globals are target-language code. The compiler infers the requested target language when a global patch includes translated code. Imports attached to patched globals or behaviors follow the same rule: omitted `language` defaults to the target language, and a mismatched language is rejected. Translated globals must avoid declaring any name listed in `guidance.compiler_owned_symbols` or `guidance.compiler_owned_bindings`. Adapter-created helper globals must also use an `adapter_global_` ID prefix, use only letters, digits, and underscores after the prefix, include target-language code, and avoid declaring compiler-owned symbols or runtime bindings.

## Imports

The compiler owns HSM/runtime imports required by generated model code. For example, Go output always gets the HSM package import it needs, and target backends add their own required runtime imports.

Frontends collect non-HSM top-level imports, then `AssignMutableImports` attributes imports used by globals or behavior bodies to those mutable regions as context for the adapter. The compiler does not try to semantically translate arbitrary library imports between languages; the adapter owns imports needed by translated globals and behavior bodies.

This gives adapters the imports they need to translate code bodies without granting authority over the model:

- behavior-specific source imports travel with the behavior as context
- global-specific source imports travel with the global as context
- adapter patches may replace/add target-language imports for translated globals and behaviors
- structural non-HSM imports stay on the program

Backends merge compiler-owned runtime imports with adapter-provided target imports when emitting target code, filtering by target language. When no adapter translates a foreign behavior or global, attributed source imports are preserved inside that region's target-language manual-porting comment instead of being emitted as executable target imports.

When an adapter returns program-level imports, the compiler treats them as the replacement set for target-language program imports only. Non-empty program-level imports are accepted only when the same patch includes translated target global or behavior code. Returning `"imports": []` explicitly clears target-language program imports. Non-target structural imports remain on `Program.Imports` as source context and are filtered out by target backends.

Adapter imports are validated twice: first within each patched program/global/behavior region, then across the full target program after all patches are applied. Import paths, aliases, defaults, named specifiers, and supported source-context hidden specifiers must be explicit values without surrounding whitespace; the compiler rejects ambiguous text rather than trimming it. Because backends hoist target imports into one file, an adapter must not bind the same local name to different imports in separate regions or to names declared by translated target-language globals. Java/Rust/Python wildcard imports and Dart `hide` combinators may be preserved from native source context, but translated target imports cannot introduce them because their implicit bindings cannot be collision-checked. Native Dart source context may preserve prefix-plus-`show` imports, but translated target imports must choose either a prefix alias or shown names so local bindings stay explicit. Compiler-owned runtime paths such as `Stateforward.Hsm`, `Stateforward.Hsm.Context`, `Stateforward.Hsm.Event`, `Stateforward.Hsm.Hsm`, `Stateforward.Hsm.Instance`, `Stateforward.Hsm.Model`, `com.stateforward.hsm`, `com.stateforward.hsm.Event`, `com.stateforward.hsm.*`, `java.lang.Object`, `hsm/hsm.hpp`, `package:hsm/hsm.dart`, `dart:async`, `dart:core`, `@stateforward/hsm`, `@stateforward/hsm.ts`, `javascript`, `typescript`, `github.com/stateforward/hsm.go`, `context`, `time`, `chrono`, `cstdint`, `string`, `datetime`, `typing`, `std`, `std::any`, `std::future`, `std::pin`, `std::time`, and `hsm` are reserved and cannot be returned by adapters under another alias. Compiler-owned runtime bindings such as `hsm`, `Behavior`, `Context`, `Kind`, `Vertex`, Dart DSL functions like `define`, `state`, `transition`, `target`, `at`, `every`, `onCall`, and `when`, JavaScript and TypeScript signature/default-return bindings `Date` and `undefined`, TypeScript signature bindings `InstanceType` and `Promise`, Java signature binding `Object`, plus generated helper types such as `Timer`, `DateTime`, `Duration`, `Future`, `Any`, and `Pin` are also reserved where generated target code relies on them; adapters should use different aliases when helper code needs another module that would collide with those names. The adapter protocol exposes these local names as `guidance.compiler_owned_bindings` so adapters can avoid collisions before returning a patch.

## Adapter Protocol

The Codex adapter receives an `AdapterProtocol` envelope:

- `version`: protocol version
- `instruction`: concise mutation rules
- `guidance`: target import shape, compiler-owned runtime imports, local bindings, generated symbols, editable fields, and forbidden fields
- `response`: exact output format, allowed patch fields, and null/unknown-field policy
- `allowed_edits`: source program import paths as context, global IDs, behavior IDs, and per-global/per-behavior source import context, including structured import bindings
- `request`: source/target languages, package name, imports, globals, read-only model context, and behaviors

Top-level imports in the request describe source context. Top-level imports in the response are target-language imports for translated target code, not permission to rewrite source imports or model structure.

The expected response is a JSON `BehaviorPatch`:

```json
{
  "imports": [{ "path": "./helpers", "language": "python" }],
  "globals": [{ "id": "global_1", "code": "def helper():\n    pass", "imports": [{ "path": "./global_helper" }] }],
  "behaviors": [{ "id": "entry_1", "body": "instance.log.append('entered')", "imports": [{ "path": "./behavior_helper" }] }]
}
```

Use `behaviors[].body` for translated statements that belong inside the compiler-owned target behavior signature. Use `behaviors[].code` only when returning a single complete target-language callable implementation that uses the compiler-owned behavior name, such as a full function/method declaration or a backend-supported callable expression. Do not append helper declarations or other top-level code after the callable; return helper code as `globals` instead. If an adapter cannot produce a target behavior implementation, omit that behavior; the compiler will preserve the untranslated source as a comment inside the generated target behavior body.

The adapter may return plain JSON or a single fenced JSON code block. Fences may be untagged or tagged as `json` case-insensitively, including common agent output such as ```` ``` json ````. Multiple fenced blocks are rejected as ambiguous, while triple backticks inside JSON string values are preserved as behavior/global code. The only supported top-level response fields are `imports`, `globals`, and `behaviors`; fields such as `models`, `states`, `transitions`, or `events` are rejected instead of ignored. Nested patch objects are strict as well:

- imports may contain only `path`, `alias`, `default`, `specifiers`, `hidden`, `static`, `deferred`, `type_only`, and `language`
- import specifiers may contain only `name`, `alias`, and `type_only`
- globals may contain only `id`, `code`, and `imports`
- behaviors may contain only `id`, `body`, `code`, and `imports`

Global and behavior `id` values are selectors for existing editable regions, not mutable structure. Unknown IDs are rejected, except adapter-created helper globals whose IDs must start with `adapter_global_` and then contain only letters, digits, and underscores.

All scalar patch fields are strings when present, and array patch fields are arrays when present; omit a field instead of returning `null`. Identity and language strings must not contain surrounding whitespace. Code/body fields may be omitted or empty for import-only patches where otherwise legal, but whitespace-only code/body fields are rejected.

Any patch referencing an unknown behavior/global ID is rejected. Adapter-created helper globals must use IDs beginning with `adapter_global_` and then only letters, digits, and underscores. Translated existing globals and every top-level declaration in helper globals must not shadow compiler-owned generated symbols such as target behavior callables, event constants/types, model bindings, or target scaffold helpers; the protocol exposes those names in `guidance.compiler_owned_symbols`. Adapters can omit globals or behaviors they cannot translate; the compiler will keep omitted behavior source as comments inside generated behavior bodies and omitted globals as comments near top-level output where manual porting is needed.
